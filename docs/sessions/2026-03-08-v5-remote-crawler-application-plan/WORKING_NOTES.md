# Working Notes – V5 Remote Crawler Application

- 2026-03-08 — User wants a comprehensive v5 plan, not implementation, with a nice remote UI, direct crawl control, large compressed downloadable bundles for time windows, and server-side browsing/reading of downloaded news.
- 2026-03-08 — Existing v5 state is fragmented: `docs/books/README.md` advertises `docs/books/v5-crawler-architecture.md`, but that file was missing before this planning pass. `docs/ROADMAP.md` only framed R-17 as an async producer-consumer rewrite, which is too narrow for the desired operator-facing remote application.
- 2026-03-08 — Existing reusable crawl backend asset: `deploy/remote-crawler-v2/multi-domain-server.js` already has status/health, crawl control, dynamic domains, gzip export, replay export, sync manifests, SSE, logs/errors, and graceful shutdown. This is the best current base for a v5 remote application.
- 2026-03-08 — Existing reusable UI shell asset: `src/ui/server/unifiedApp/` already provides a shell app model with multiple sub-apps and operator-oriented panels.
- 2026-03-08 — Existing reusable content-browsing assets: `src/ui/server/dataExplorerServer.js` already exposes `/articles` and article detail pages; `src/ui/server/articleViewer/server.js` already exposes a cleaner standalone article library/reader.
- 2026-03-08 — Existing reusable operator-observability assets: `src/ui/server/crawlObserver/server.js`, `src/ui/server/crawlerMonitor/server.js`, and unified-app download evidence endpoints can be folded into a v5 control center instead of rebuilt from scratch.
- 2026-03-08 — Planning decision: v5 should be framed as productizing and consolidating the current remote crawler/UI/export pieces, not replacing the crawler engine first.
- 2026-03-08 — Planning decision: the v5 remote UI should be a browser-accessible shell with at least five modules: Control Room, Article Library, Bundle Manager, Monitoring, and Settings/Auth.
- 2026-03-08 — Planning decision: large export/download should become an async bundle-job system with manifests and downloadable compressed archives, rather than relying only on direct request/response gzip streams.
- 2026-03-08 — Planning decision: remote browsing/reading should explicitly reuse the existing Data Explorer `/articles` and Article Viewer concepts so the remote server is not just a crawler daemon.
- 2026-03-08 — Process repair: `docs/sessions/SESSIONS_HUB.md` referenced `docs/sessions/long-term/lt-001-advanced-crawler-ui/` as the active long-term session, but the directory was missing. This pass restores that long-term anchor so future implementation sessions have somewhere to accumulate progress.
- 2026-03-08 — Expanded planning scope from "remote app slice" to "v5 for all of it" by creating a dedicated plan directory under `docs/plans/2026-03-v5-remote-crawler-application/` with a master plan, workstreams, session map, human-review comments, AI suggestions, and an explicit testing/perfection strategy.
- 2026-03-08 — Tightened the plan set to make three product properties explicit rather than implied: always-on remote hosting, fast operator UI under load, and separation between the UI/gateway event loop and heavy crawl/bundle execution.
- 2026-03-09 — User confirmed that more aspirations should become first-class requirements; promoted auth, restart-safe runs/jobs, bundle integrity/retry behavior, host-protection guardrails, and explicit responsiveness budgets into the v5 plan set.
- 2026-03-09 — User also wants intelligent crawling using place and topic hub guessing to be core. Updated the v5 architecture and execution docs to make hub guessing part of the main product path, grounded in existing assets such as `CountryHubGapAnalyzer`, `PlaceHubPatternLearningService`, and the place/topic hub guessing UI query surfaces.
